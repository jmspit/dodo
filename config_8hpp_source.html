35  /**
36  * Singleton class representing the deployment configuration, combining deployment constants from the configuration
37  * file, environment variables (whose names start with the appname) and some operating system quantities useful to
38  * developers, like the CPU count or an imposed memory limit.
39  *
40  * Configuration files are YAML files. Configuration values may use the encryption format of the DataCrypt
41  * interface, so that secrets can safely be stored in YAML configuration files.
42  *
43  * @code
44  * service:
45  * fqdn: remote.server.org
46  * port: 1827
47  * client_id: ENC[]
48  * client_secret: ENC[]
49  * @endcode
50  *
51  * Each Application will have only one configuration file, and its location must be known when the singleton
52  * Config instance is created, developers need to call initialize() first. All subsequent calls to getConfig() will
53  * return the same pointer without the need to specify the configuration file path each time.
54  *
55  *
56  * The Config class exposes the configuration data as const YAML::Node reference. So to modify or
57  *
58  * As multiple threads may be reading and modifying the configuration data, access must be serialized. However, as the
59  * data is exposed as a reference to a YAML::Node, this class cannot control serialization transparently. Developers
60  * know when they are done reading or writing the configuration data. For example
61  *
62  * @code
63  * {
64  * threads::Mutexer( getConfig()->getMutex() );
65  * const std::string username = config["username"].as<std::string>();
66  * }
67  * // Mutexer is out of scope, destructed and the getConfig()->getMutex() lock is released.
68  * @endcode
69  */
