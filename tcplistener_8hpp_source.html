95  /**
96  *
97  * The TCPListener listens, accepts connections and generates socket events to produce TCP work to a pool of
98  * TCPServer worker objects. Developers must implement a protocol by subclassing TCPServer. The produced
99  * by a TCPListener is either
100  *
101  * - handling new connections (and optionally perform a protocol handShake).
102  * - reading data from sockets and sending responses
103  * - closing down connections on hangups or errors.
104  *
105  * The developer sub-classes TCPServer to implement a request-response protocol.
106  *
107  * TCPListener uses the Linux epoll_wait interface to pick up socket events and distribute the work to TCPServers.
108  *
109  * The TCPListener is started start( TCPServer *server ) with an initial TCPServer object. The
110  * TCPServer::addServer() method is used to spawn additional TCPServer objects up to Params.minservers when
111  * TCPListener::run() is invoked. The TCPListener will spawn TCPServer worker threads if the amount of pending
112  * work exceeds the number of existing TCPServer objects, up to Params.maxservers.
113  *
114  * The TCPServer descendant class implements the details of handShake, readSocket and shutDown, and are each
115  * passed a SocketWork that needs to be serviced -a SocketWork specifies the socket and the event state it is in.
116  *
117  * Sockets are set to non-blocking mode, and TCPServer::readSocket is not guaranteed to see a full request, data
118  * might arrive in multiple chunks.
119  *
120  * For a given connection, a TCPServer descendant will cycle through
121  *
122  * - a handShake call
123  * - zero or more readSocket calls
124  * - a shutDown call
125  *
126  * On connection errors and hangups call to shutDown will follow - the BaseSocket object passed to shutDown() is
127  * not destroyed yet, allowing TCPServer implementations to clean up failed connections.
128  */
